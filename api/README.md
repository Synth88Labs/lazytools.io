# /api — the entire LazyTools backend

Two PHP files. That's the whole server side of LazyTools, and it exists only for **anonymous tool
ratings**. Tools themselves never talk to any server (see [/how-it-works/](https://lazytools.io/how-it-works/)).

| File | Purpose |
|---|---|
| `rate.php` | `POST {"tool", "stars"}` → one row. Stores tool slug + stars + timestamp. **No IP, no cookies, no UA, no identifiers.** |
| `stats.php` | `GET` → aggregate `{tool: {count, avg}}`, only for tools with ≥25 ratings (the display threshold). |
| `tools-allowlist.json` | Regenerated by `npm run build` from the tool registry — `rate.php` rejects slugs that aren't real tools. |
| `data/` | SQLite database (`ratings.db`), auto-created on first rating. Blocked from the web by `data/.htaccess`. |

## Deployment

Fully automatic: `scripts/postbuild.mjs` copies the API into `dist/api/`, so every site deploy ships it.
No database server, no credentials, no cPanel setup — storage is a zero-config SQLite file that PHP's
built-in `pdo_sqlite` creates on first use. Deploys never touch `ratings.db` (the FTP sync only manages
files it uploaded itself).

Before each CI build, `scripts/fetch-ratings.mjs` pulls `/api/stats.php` into `src/data/ratings.json`
so pages can render rating averages and `aggregateRating` JSON-LD for tools past the 25-rating threshold.

## Verify after deploy

```
curl -X POST https://lazytools.io/api/rate.php -H "Content-Type: application/json" \
     -d '{"tool":"units/kg-to-lbs","stars":5}'          # → {"ok":true}
curl https://lazytools.io/api/stats.php                  # → {} until thresholds are met
curl -I https://lazytools.io/api/data/ratings.db         # → 403 (blocked)
```

Abuse posture: a localStorage flag client-side softens duplicate votes; the ≥25 display threshold makes
low-effort manipulation pointless. We deliberately accept some noise rather than fingerprint users.
