# Browser-Tool Market Research — 2026-07-07 (High-Demand / Low-Competition Gap Scan)

## Executive summary

This run hunted exclusively for winnable gaps: high demand with weak, upload-based, or platform-locked competition. Three candidates cleared the bar. The clear #1 is a **Poland KSeF FA(3) invoice viewer + validator** — the third run of the twice-proven e-invoicing playbook, with the largest population yet (every Polish VAT business since April 2026, micro-entrepreneurs by January 2027, penalties up to 100% of VAT from January 2027) and only new, low-authority micro-sites competing in English. #2 is a **client-side PDF accessibility quick-checker** (EAA-driven; every free checker today is Windows-only desktop or upload-based). #3 is a **PDF redaction suite (rasterizing redactor + redaction checker)**, boosted by the December 2025 Epstein-files redaction failures and a study showing 65% of "redacted" agency PDFs still leak text. UAE PINT AE is a strong scheduled follow-up (pre-build ~Oct–Nov 2026 ahead of the Jan 2027 go-live). WhatsApp export viewer and systemd timer tools were checked and found already served by good client-side incumbents.

## Opportunities

### 1. KSeF FA(3) Invoice Viewer + Validator (Poland) — 23/25

- **Proposed category/slugs:** `/file/ksef-viewer/`, `/file/ksef-validator/`; enhance existing `/file/e-invoice-viewer/` with FA(2)/FA(3) auto-detection; optional cheap variant `/file/peppol-bis-viewer/` (Belgium, distinct query, existing UBL parser already handles it).
- **Scores:** Demand 5 · Feasibility 5 · Competition gap 3 · Durability 5 · Fit 5.
- **Future driver (named, dated):** Poland's KSeF clearance mandate — mandatory issuance for large taxpayers (>PLN 200M) since **1 Feb 2026**, all VAT-registered businesses since **1 Apr 2026**, micro-entrepreneurs from **1 Jan 2027**; *receiving* capability mandatory for everyone since Feb 2026. 2026 is a penalty grace year; from **Jan 2027 penalties reach up to 100% of the VAT amount** — demand grows into 2027. Every invoice must be FA(3) XML (schema replaced FA(2) in Jan 2026); an invoice does not legally exist until KSeF accepts it.
- **Evidence:**
  - Timeline/penalties/FA(3) schema: https://www.vatupdate.com/2025/12/30/mandatory-e-invoicing-in-poland-ksef-a-complete-guide-to-2026-deadlines/ ; https://evotax.pl/en-us/blog/national-e-invoice-system-ksef-in-poland/ ; https://edicomgroup.com/blog/poland-will-make-b2b-electronic-invoicing-mandatory ; https://ec.europa.eu/digital-building-blocks/sites/spaces/DIGITAL/pages/467108896/eInvoicing+in+Poland
  - Competition observed: ksef-pl.com (new client-side micro-site with viewer/validator), ksefgpt.pl (validator/generator, Polish-focused), github.com/ksef4dev/ksef-fop (developer library, not end-user). No established brand owns "KSeF XML viewer" in English SERPs; incumbents are weeks–months old with low authority. (Observed: incumbents exist; inference: the query space is winnable, unlike Smallpdf-class saturation.)
- **Feasibility notes:** Pure DOMParser XML mapping — same architecture as the shipped e-invoice viewer. FA(3) is Poland's own XSD (Podmiot1/Podmiot2 parties, `Fa` header, `FaWiersze` line items, VAT summary), not UBL/CII, so a new mapping layer is needed but no new tech. Validator MVP = deterministic rule checks: required fields, NIP checksum (weighted mod-11, fully client-side), totals consistency, FA(2)-vs-FA(3) version detection — honestly labeled "structure check, not official MoF validation." No server, no live data.
- **AI-resistance:** deterministic output, privacy-sensitive input (real invoices with counterparty data — exactly what users shouldn't paste into chatbots), zero-friction, repeat weekly workflow — 4/5.
- **Suggested MVP tool set:**
  1. `/file/ksef-viewer/` — drop/paste FA(3) XML → human-readable invoice card (seller/buyer with NIP, invoice number/date/currency/payment, line items, VAT breakdown), print / save-as-PDF.
  2. `/file/ksef-validator/` — rule-based structure check with pass/fail list and field locations; NIP checksum built in (not a standalone page — spam-compliance).
  3. FA(3) auto-detect + route in the existing e-invoice viewer.
  4. (Optional, ~1 hr) `/file/peppol-bis-viewer/` landing variant — Belgium's B2B mandate live since 1 Jan 2026, penalties enforced since 1 Apr 2026, format is Peppol BIS 3.0 / UBL 2.1 which the existing parser already reads.
- **Blog/SEO angle:** "KSeF 2026/2027: how to open and read an FA(3) invoice XML — no software, nothing uploaded." Deadline/penalty table (Feb 2026 / Apr 2026 / Jan 2027 / penalties), FA(3) field-by-field walkthrough, FA(2)→FA(3) differences, privacy argument, 5–8 FAQs. Long-tail: "open ksef xml file", "faktura ksef xml podgląd", "FA(3) invoice format explained", "ksef nip validation".

### 2. PDF Accessibility Quick-Checker (client-side, EAA-driven) — 21/25

- **Proposed category/slug:** `/pdf/accessibility-checker/`.
- **Scores:** Demand 4 · Feasibility 4 · Competition gap 4 · Durability 4 · Fit 5.
- **Future driver:** European Accessibility Act in force since **28 June 2025** — customer-facing PDFs (invoices, statements, contracts, manuals) must meet EN 301 549 / WCAG 2.1 AA, effectively tagged PDF / PDF/UA; transition for existing services runs to **28 June 2030**, so enforcement demand grows for years. (Note: the earlier-rejected *website* accessibility scanner failed on cross-origin fetching; a *file-input* PDF checker has no such problem — distinct tool.)
- **Evidence:**
  - EAA scope incl. PDFs and 2030 transition: https://pdfix.net/pdf-compliance/european-accessibility-act/ ; https://www.levelaccess.com/compliance-overview/european-accessibility-act-eaa/ ; https://accessible-eu-centre.ec.europa.eu/content-corner/news/eaa-comes-effect-june-2025-are-you-ready-2025-01-31_en
  - Competition all desktop or upload: PAC is **Windows-only** (https://pac.pdf-accessibility.org/en); PDFix Desktop Lite is a desktop install and self-describes as "the only free PDF accessibility checker available for macOS and Linux" (https://pdfix.net/products/pdfix-desktop-lite/); axesCheck is web-based but processes on their servers (https://check.axes4.com/en). Observed Mac-user frustration on Adobe's feedback forum about PC-only checkers (https://acrobat.uservoice.com/forums/590923-acrobat-for-windows-and-mac/suggestions/40593796). **No fully client-side browser checker was found** — the strongest competition-gap axis of this run.
- **Feasibility notes:** pdf.js — `page.getStructTree()` for tag tree, catalog `MarkInfo/Marked`, document `Lang`, `Title` + `DisplayDocTitle`, XMP PDF/UA identifier, figure alt-text presence, table structure presence, font embedding. Honest MVP = "quick check" of the machine-verifiable basics (tagged? language? title? alt text? bookmarks?), explicitly *not* full Matterhorn/PDF/UA certification, with a pointer that certification needs PAC. Moderate complexity but no wasm needed.
- **AI-resistance:** binary files, deterministic checks, privacy-sensitive (unpublished contracts/reports), repeat compliance workflow — 4–5/5.
- **MVP tool set:** single checker page with a pass/warn/fail checklist, per-check explanation, and "what to fix in Word/Acrobat before export" guidance.
- **Blog/SEO angle:** "Is your PDF EAA-compliant? Check it in your browser (works on Mac — no PAC needed)." Long-tail: "PDF accessibility checker mac free", "check if PDF is tagged", "EAA PDF requirements 2026".

### 3. PDF Redaction Suite: Rasterizing Redactor + Redaction Checker — 20/25

- **Proposed category/slugs:** `/pdf/redact-pdf/` + `/pdf/redaction-checker/`.
- **Scores:** Demand 4 · Feasibility 4 · Competition gap 3 · Durability 4 · Fit 5.
- **Future driver:** structural privacy driver plus a fresh news catalyst — the **December 2025 Epstein-files release exposed recoverable text under "redactions"** (https://pdfa.org/a-case-study-in-pdf-forensics-the-epstein-pdfs/ ; https://allaboutpdf.com/blog/2025/12/23/epstein-files-redaction-fail), and an academic study of ~40,000 PDFs from 75 security agencies found **65% of claimed-redacted files still exposed hidden information** (https://www.argeliuslabs.com/deep-research-on-pdf-redaction-failures-and-security-risks-exploits-and-best-practices/). Users redact precisely the documents (bank statements, IDs, contracts) they should never upload — Smallpdf/Xodo/Redactable all upload (https://smallpdf.com/redact-pdf ; https://xodo.com/blog/how-to-redact-pdf-without-adobe).
- **Competition caveat (why gap = 3):** client-side micro-site redactors exist (just-redact.com, onlineredactorpdf.com with MuPDF wasm, customizedpdf.com) — new, low-authority, single-tool sites. The *checker* niche ("is my PDF properly redacted?") has essentially one weak incumbent (textfixer.com's unredact tester) — that half is the wedge.
- **Feasibility notes:** Redactor: pdf.js render → canvas black-out → rebuild via pdf-lib/jsPDF (rasterize-and-flatten = provably unrecoverable text; disclose the size/text-layer trade-off). Checker: pdf.js text extraction under user-drawn or detected black rectangles + hidden-metadata scan (XMP, annotations, attachments) — pairs with the existing image-metadata-remover.
- **AI-resistance:** binary files, deterministic, maximal privacy sensitivity, zero-friction — 4/5.
- **Blog/SEO angle:** "The Epstein files redaction failure: why black boxes don't redact — and how to check your own PDFs." Long-tail: "check if pdf redaction worked", "redact bank statement without uploading", "text under black box pdf".

- **South Asia land-area unit cluster (bigha/katha/dhur/ropani/aana/marla/kanal/gaj/guntha/cent/decimal)** — 18/25 nominal but competition-gap axis 2 blocks: multiple functioning incumbents observed (landeed.com, assetyogi.com, ropanibigha.com with district-specific tables, allinonecalc.online). Real complexity moat exists (state-by-state variance: Bihar bigha 27,220 sq ft vs Nepal bigha 72,900 sq ft). Promote with keyword-gap proof of winnable state-level long-tail; natural companion to the BS⇄AD converter audience.
- **WhatsApp chat export viewer (.txt/.zip → chat UI, search, stats)** — 16/25. Privacy angle is real (most viewers upload) but a client-side incumbent already claims browser-only processing (wachatviewer.com) and my demand searches returned thin recurring-pain evidence. Promote with observed forum pain threads.
- **National ID checksum validator cluster (NIP/PESEL/codice fiscale/NIF/Aadhaar-format)** — unscored; surfaced by lane 3 logic (privacy-sensitive, deterministic, users shouldn't paste IDs into chatbots) but no demand/competition evidence gathered this run. NIP validation ships inside the KSeF validator regardless. Needs a dedicated scan.

## Rejected today

- **systemd OnCalendar expression tool** — systemd.guru already is the crontab.guru of systemd; niche taken.
- **Spain Verifactu tooling** — requires AEAT-certified software with hash-chained real-time submission (server-side by definition); deadline slipped to Jan/Jul 2027 (Royal Decree 15/2025); no client-side role.
- **Belgium Peppol viewer as standalone bet** — existing e-invoice viewer already parses UBL 2.1/Peppol BIS 3.0; folded into Opportunity #1 as an optional variant landing page only.
- **KSeF PDF→XML converter** — requires OCR/AI extraction; fails determinism and quality bar (ksefgpt.pl offers it; let them).

## Lane notes

1. **Platform & API signals:** not the focus of this gap run; no new checks beyond pdf.js `getStructTree` feasibility (stable). JPEG XL/Chrome watchlist item unchanged.
2. **Format & ecosystem shifts:** WhatsApp export format checked (incumbent found); passkey export (CXF) judged too early to search-verify; FA(3) XML is the format win of the run.
3. **Demand & pain signals:** Reddit-scoped searches underperformed (two returned zero results); pivoted to bank-statement/redaction privacy angle which yielded the strongest new pain signal (Epstein redaction failures, 65%-leak study).
4. **Competitor & gap scan:** ksef-pl.com / ksefgpt.pl (new, low-authority), PAC/axesCheck/PDFix (desktop or upload), Smallpdf/Xodo redaction (upload), wachatviewer (client-side, taken), systemd.guru (taken).
5. **Regulatory & institutional drivers:** Poland KSeF (Feb/Apr 2026, Jan 2027 penalties) — build now; Belgium (live, penalties since Apr 2026) — variant page; Spain Verifactu — rejected (server-bound, delayed); UAE PINT AE (Jan 2027) — scheduled pre-build; EAA (June 2025 → 2030 transition) — Opportunity #2.
